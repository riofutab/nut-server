# v0.1.2 Follow-up Plan

> For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the three shutdown recovery regressions found after the v0.1.1 release candidate so reconnects, timeout repair, and UPS-trigger latching all behave correctly.

**Architecture:** Keep the current persisted shutdown-state design, but tighten the ownership rules around active execution and replay. The slave needs to report the terminal result on the latest live session, and the master needs to separate "command finished" from "UPS auto-trigger latch still active" so one low-battery event does not create repeated shutdown broadcasts.

**Tech Stack:** Go, TCP/TLS control channel, persisted JSON state files, Go test suite

---

### Task 1: Rebind in-flight slave shutdown execution to the latest session

**Files:**
- Modify: `internal/slave/client.go`
- Test: `internal/slave/client_test.go`

- [ ] Add connection-session tracking for commands that are already executing so a replay on a new TCP connection can update where the terminal ACK is sent.
- [ ] Write a regression test that simulates: shutdown starts, old session drops, replay arrives on a replacement session, execution finishes, and the final `executed` or `failed` ACK is emitted on the replacement session.
- [ ] Update the execution path so "already executing" does not mean "stop reporting"; it should mean "reuse the running work and rebind final reporting to the newest live session".
- [ ] Run: `go test ./internal/slave -run "TestHandleShutdown.*|Test.*Replay.*" -v`

### Task 2: Keep UPS auto-shutdown latched until reset

**Files:**

- Test: `internal/master/server_test.go`
