# v0.1.2 Recovery And Release Design

**目标**

`v0.1.2` 同时解决两类问题：

1. 修复当前 shutdown 恢复流程里的 3 个正确性问题，避免断线、超时和 UPS 持续异常时出现错误行为。
2. 在保留现有完整发布包的前提下，新增更适合真实部署的独立安装包、升级包、快速安装脚本和在线安装入口。

---

## 1. 范围

本次设计包含：

- 修复 slave 执行关机时断线重连后最终状态上报丢失的问题
- 修复 master 在 UPS 条件持续异常时重复自动触发 shutdown 的问题
- 修复节点 timeout 后重连无法纠正最终状态的问题
- 保留当前 full release 包
- 新增 master 独立安装包
- 新增 slave 独立安装包
- 新增已安装机器的一键升级包
- 新增更省参数的离线安装脚本
- 新增在线安装入口脚本
- 更新 GitHub Release 工作流和 README

本次不包含：

- 管理接口鉴权
- 新的监控或 metrics 体系
- 自动回滚机制
- Windows 安装支持

---

## 2. 当前问题

### 2.1 运行时问题

当前 slave 在 `executeShutdown` 执行期间如果连接断开，最终 `executed` / `failed` ACK 仍然会尝试发到旧连接。即使新连接已经建立，master 也收不到真实最终状态，最后会把节点记成 `timeout`。

当前 master 在命令完成后会立刻清掉自动触发状态。只要 UPS 条件还没有恢复，下一次轮询又会再触发一次新的 shutdown。结果是同一次电源事件会被反复广播。

当前 master 对 timeout 后重连的节点不会再 replay shutdown。slave 又只有在收到 replay 时才会重新上报持久化的最终状态，因此 timeout 后的真实结果无法纠正回 master。

### 2.2 发布和部署问题

当前 release 只有“完整包”一种形式，虽然能用，但对真实部署不够方便：

- 只安装 master 的机器，仍然会拿到 slave 二进制和多余文件
- 只安装 slave 的机器，同样会拿到不需要的内容
- 已安装机器升级时没有单独升级包，只能手工判断要替换什么
- 现在的安装脚本参数偏多，常见场景下输入成本高
- 虽然已经能通过 GitHub Release 发包，但还没有在线安装入口

---

## 3. 总体方案

### 3.1 版本目标

`v0.1.2` 作为“稳定性 + 交付体验”版本，一次性交付：

- 更可靠的 shutdown 恢复逻辑
- 更清晰的发布产物矩阵
- 更顺手的离线安装和在线安装入口
- 面向已安装机器的升级包

### 3.2 设计原则

- 先保证控制流正确，再增强交付体验
- 保留当前可用路径，不强推用户迁移
- 快捷脚本只是包装层，不替代底层细粒度脚本
- 升级默认保守，不覆盖用户配置和状态数据
- 发布产物命名清晰，用户仅通过文件名就能知道该下哪个包

## 4. 运行时修复设计

### 4.1 Slave 执行中断线后要把最终结果发到新连接

#### 当前问题

当某个 `command_id` 已经处于 `executing`，replay 到新连接后只是检测到“已经在执行”，然后直接返回。原来的执行 goroutine 仍然持有旧 `session`，最终状态继续往断开的连接发送。

#### 设计

slave 需要把“执行中的关机任务”与“当前用于回 ACK 的连接”拆开管理。

新增一个按 `command_id` 维护的执行上下文，至少包含：

- 当前是否已有真实关机在执行
- 当前应使用哪个 `session` 发送最终 ACK
- 最近一次 shutdown message 的必要上下文

行为改成：

1. 第一次收到 shutdown 时，创建执行上下文，绑定当前 `session`，启动真实执行。
2. 如果执行期间断线并收到 replay：
   - 不再重复启动真实关机；
   - 更新该 `command_id` 对应的“当前 ACK session”为新连接；
   - 如有必要，立即把 `executing` 状态再发一次给新连接。
3. 当真实执行结束时，从执行上下文中取“最新 session”发送最终 `executed` / `failed` ACK。
4. 如果最终 ACK 发送失败，仍然保留持久化状态；后续只要再次收到 replay，就能继续补发。

#### 结果

真实关机只执行一次，但最终结果会尽量发到最新可用连接，避免“执行已经完成但 master 一直看不到”的问题。

### 4.2 Master 自动触发要保持 latch，直到人工 reset

#### 当前问题

当前 `commandComplete()` 后会直接清掉 `shutdownIssued` / `activeCommand`，这会让 UPS 仍然异常时继续自动触发新的 shutdown。

#### 设计

把“当前命令是否完成”和“UPS 自动触发 latch 是否仍然生效”分开。

新增一个独立的 auto-shutdown latch 状态，表示：

- 最近一次由 UPS 条件触发的自动关机已经触发过
- 在人工 reset 之前，不允许再次自动触发

行为改成：

1. UPS 条件首次满足时，创建 shutdown 命令并置位 auto-shutdown latch。
2. 命令完成后，只清理当前活动命令，不清理 auto-shutdown latch。
3. 后续轮询即使 UPS 条件仍异常，也不再自动生成新命令。
4. 只有在管理员调用 reset 时，才清掉 latch，允许下一次 UPS 事件重新触发。
5. 手工触发 shutdown 不依赖这个 latch，仍然按当前逻辑工作。

#### 结果

一次 UPS 事件只会触发一次自动 shutdown，直到人工确认并 reset。

### 4.3 Timeout 后重连仍然允许修正最终状态

#### 当前问题

master 现在对超时后的节点直接停止 replay，导致 slave 没机会把持久化的最终状态重新发回来。

#### 设计

replay 规则不能简单用“是否已经过 timeout 时间”来截断，而要区分两件事：

- 这个节点是否已经有真实最终状态
- 这个节点是否还需要补报状态

行为改成：

1. master 对 reconnect 节点判断：
   - 如果节点还没有 `executed` / `failed`，即使当前汇总状态已经是 `timeout`，也允许 replay。
   - 只有在节点已经有真实最终状态时，才不需要 replay。
2. slave 收到 replay 后：
   - 如果本地持久化里已经是 `executed` / `failed`，立即把最终状态重发给 master；
   - 如果本地仍是 `accepted` / `executing`，继续按恢复逻辑执行或补报。
3. master 保持当前“真实最终状态可以覆盖 timeout”的规则不变。

#### 结果

节点即使在 timeout 之后才恢复连接，也仍然可以把 master 上的 `timeout` 改正为真实最终状态。

## 5. 发布产物设计

### 5.1 保留完整包

继续保留当前 full package，面向第一次接触项目或需要同时拿到所有内容的用户。

建议命名：

- `nut-server_<version>_linux_amd64.tar.gz`
- `nut-server_<version>_linux_arm64.tar.gz`

内容保持为：

- `bin/nut-master`
- `bin/nut-slave`
- `configs/`
- `systemd/`
- `scripts/`
- `README.md`
- `LICENSE`

### 5.2 新增 master 独立安装包

面向只部署 master 的主控机。

建议命名：

- `nut-master_<version>_linux_amd64.tar.gz`
- `nut-master_<version>_linux_arm64.tar.gz`

内容只包含：

- `bin/nut-master`
- `configs/master.example.yaml`
- `systemd/nut-master.service`
- `scripts/install-master.sh`
- `scripts/quick-install-master.sh`
- `scripts/upgrade-master.sh`
- `README-master.md` 或主 README 中的角色说明片段
- `LICENSE`

### 5.3 新增 slave 独立安装包

面向只部署 slave 的被控机。

建议命名：

建议命名：

- `bin/nut-master`

## 6. 安装和升级脚本设计

### 6.1 保留底层安装脚本

现有：

- `install-master.sh`
- `install-slave.sh`

继续保留，作为最完整、最细粒度的安装入口。

### 6.2 新增快速离线安装脚本

新增：

- `quick-install-master.sh`
- `quick-install-slave.sh`

定位：

- 面向最常见部署路径
- 参数比底层脚本更少
- 通过合理默认值减少手工输入

建议行为：

#### quick-install-master.sh

最少输入：

- `--token`
- `--snmp-target`

可选输入：

- `--listen-addr`
- `--snmp-community`
- `--disable-tls`
- `--tls-cert-file`
- `--tls-key-file`
- `--tls-ca-file`
- `--tls-require-client-cert`

默认行为：

- 默认写入标准路径
- 默认启用 systemd 并启动
- 默认创建数据目录

#### quick-install-slave.sh

最少输入：

- `--node-id`
- `--master-addr`
- `--token`

可选输入：

- `--tags`
- `--disable-tls`
- `--tls-ca-file`
- `--tls-server-name`
- `--tls-cert-file`
- `--tls-key-file`
- `--tls-insecure-skip-verify`

默认行为与 master 快速脚本保持一致。

### 6.3 新增升级脚本

新增：

- `upgrade-common.sh`
- `upgrade-master.sh`
- `upgrade-slave.sh`

#### 升级原则

- 停服务
- 备份旧二进制和旧 systemd 文件
- 不覆盖 `/etc/nut-server/*.yaml`
- 不触碰状态文件和运行数据
- 替换程序和脚本
- 刷新 systemd
- 启动服务
- 输出升级前后版本信息或文件时间

#### 升级前备份

建议备份目录：

- `/var/backups/nut-server/<timestamp>/`

建议备份内容：

- 已安装二进制
- 现有 systemd unit
- 现有安装/升级脚本

#### 升级失败策略

如果替换后启动失败，脚本至少要：

- 明确打印失败阶段
- 提示备份目录位置
- 提示如何手工恢复

本次不做全自动回滚。

### 6.4 在线安装入口

新增在线安装脚本，例如：

- `install-online.sh`

定位：

- 一条命令从 GitHub Release 拉取对应版本和架构的 tar.gz
- 解压到临时目录
- 调用包内的快速安装脚本

建议用法：

- 安装 master
- 安装 slave
- 指定版本
- 默认取 latest release

在线脚本只做下载、校验、解压和转调，不把真实安装逻辑重复写一遍。

## 7. 发布流程设计

### 7.1 构建阶段

`build-linux.sh` 继续负责基础 Linux 构建，但输出目录结构要能被多个打包函数复用。

建议继续先生成：

- `dist/linux-amd64/`
- `dist/linux-arm64/`

然后由 release 打包脚本从这些构建结果中裁剪出不同产物。

### 7.2 release 打包脚本

现有 `package-release.sh` 只生成 full package。需要改造成：

- 支持 full package
- 支持 master package
- 支持 slave package
- 支持 upgrade package

建议把脚本拆成更清晰的打包函数：

- `package_full`
- `package_master`
- `package_slave`
- `package_upgrade`

每个函数再按架构生成对应 tar.gz。

### 7.3 GitHub Release 工作流

`release.yml` 继续在 tag push 时运行，但最终上传的文件不再只有 full package，而是所有发布产物。

Release 附件至少包括：

- full package x 2
- master package x 2
- slave package x 2
- upgrade package x 2

如果后续还要加校验文件，可以再扩展 `.sha256`。

## 9. 测试和验证

### 9.1 运行逻辑

需要新增回归测试覆盖：

- slave 执行中断线后，最终 ACK 发到新连接
- UPS 持续异常时不会重复自动触发 shutdown
- timeout 后重连仍可把状态从 `timeout` 改成 `executed` / `failed`

### 9.2 配置与 TLS

已有 TLS 测试继续保留，新的快速安装脚本和在线安装脚本不能破坏：

- `--disable-tls`
- `tls.disabled`
- 证书必填校验

### 9.3 打包与脚本

需要验证：

- full/master/slave/upgrade 四类包都能生成
- 包内文件结构正确
- 快速安装脚本可正常调用底层脚本
- 升级脚本不会覆盖配置
- 在线安装脚本能正确拼出下载地址并调用对应脚本

### 9.4 最终验收

最终至少要通过：

- `go test ./...`
- `go build ./...`
- `VERSION=v0.1.2 ./scripts/package-release.sh`
- 检查 `dist/release/` 下所有新包是否齐全

## 10. 风险与控制

### 风险 1：slave 执行上下文改动引入重复关机

控制方式：

- 明确区分“真实执行是否已经启动”和“当前 ACK 应发到哪个 session”
- 用回归测试锁定“只执行一次，但可多次补报状态”

### 风险 2：UPS latch 改动影响手工 reset

控制方式：

- 明确把 auto-shutdown latch 与 active command 分离
- 增加 reset 相关测试，确保 reset 后下一次 UPS 事件仍可触发

### 风险 3：升级脚本误覆盖用户配置

控制方式：

- 升级脚本不写配置文件
- 默认仅替换二进制、systemd、脚本
- 先备份后替换

### 风险 4：在线安装脚本和离线逻辑分叉

控制方式：

- 在线脚本只负责下载和转调
- 真实安装逻辑全部落在包内脚本里

---

## 11. 实施顺序

建议顺序：

1. 先修 3 个运行时正确性问题并补测试
2. 再重构 release 打包脚本，产出四类包
3. 再补快速安装脚本和升级脚本
4. 最后补在线安装入口和 README

这样可以先锁定稳定性，再扩展交付能力，避免把有问题的控制流继续包装成更多安装形式。
